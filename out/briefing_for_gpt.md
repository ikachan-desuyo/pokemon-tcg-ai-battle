- 方針：勝率A/Bは「ミラー新旧対決＋対フィールド」両方で測る。勝手な提出/コミットはしない運用。

## 2. プール固有の事実（このレギュ特有）
- [Fact] **イグニッションエネルギー**：特殊エネ1枚＝無色3。**進化ポケモンのみ有効・番末トラッシュ**＝**非常に高効率な加速カード**。
- [Fact] **メガポケモンexはKO時3サイド**（通常exは2、たね1）。**弱点は被ダメ2倍**。
- [Fact] 約2000枚プール。現実のポケカとはカード/レギュが異なり、**現実の対戦ログ・「理想手」データは流用不可**。

## 3. 提出デッキ＝MegaStarmie（運用中）
- [Fact] メガスターミーex（水・**弱点雷**・HP330・1進化）。イグニで2ターン目にNebula Beam 210（=**弱点も“相手にかかっている効果”も計算しない**）。
- [Measure] **現在のシミュレーション環境では最有力候補**：対フィールド≈0.90、提出スコア776.4。
- 理由（Nebula＋イグニの性質による傾向）：
  - **Nebulaの性能により、耐久・減ダメ・状態異常等の妨害戦略が機能しにくい場面が多い**（効果無視のため）。
  - イグニが揮発（使用後トラッシュ）→ 特殊エネ破壊系の妨害が刺さりにくい。
  - 2ターン目210の速度で遅い2進化デッキに先んじやすい。
- [Measure] ラダー実#1の**Archaludon(ブリジュラスex)**を忠実再現(ジーランス＋カキツバタ＋ポケパッドのエンジン)しても、**vs 我々のMegaStarmie=0.32**。＝**simでは実#1構成でもMegaStarmieに届かない**。

## 4. 構築した「三位一体」AI基盤（役割分担／全コミット済み）
**実行時AI(提出用)＝①ヒューリスティック / 開発支援AI＝②LLMコーチ / 学習AI＝③価値関数**。

### ① 高速ヒューリスティックbot（提出本線・DeckBot＋専用bot群）
- 設計：**カード毎の「目的ゲート」**（ボス=KO対象時のみ／夜タンカ=回収対象時のみ／いれかえ=攻撃役を前に出す時のみ／ジャッジ=自分が損せず相手手札が多い時のみ 等）＋**結果志向スコア**（掘り札を「使った後に何が可能になるか」で評価＝depth-1近似）。
- [Measure] Archaludon専用botの作り込みで **vs MegaStarmie 0.205→0.35、対フィールド 0.48→0.58**。可変火力技も正しく評価（例:レイジングハンマー=80+被ダメ）。
- [Fact] **カード単位の改善は概ね反映済み**。一括温存・素朴な先読みはA/Bで悪化（このアーキの伸び代は限定的）。

### ② LLMコーチ（LLM-in-the-loop開発）
- `structured_trace.py`(対戦→機械可読JSON) / `decision_trace.py`(**全候補手＋score＋ゲート却下理由＋なぜ選んだか**を可視化) / `coach.py`(**知識ベース＋トレースをClaudeに渡し構造化改善提案**)。
- [Fact] 実行時botは高速のまま＝制約非依存。ループ実証済み（**有効な提案を採用・無効を却下する規律**が機能）。

### ③ 学習価値関数（盤面ベクトル化＝AlphaZero型の芽）
- `value_model.py`：盤面を**固定長16次元ベクトルに符号化**→自己対戦の(状態,最終勝敗)からロジ回帰で**価値モデル(状態→勝率)**を学習。
- [Measure] **勝敗予測 精度0.627 / AUC0.619**（基準=サイド差符号のみ0.513。2359状態、80/20分割）。重みは解釈可能（has_active/prize_diff/opp_hpは妥当、opp_board_energyはデータ偏重の交絡）。
- `value_bot.py`(ValueModelBot)：**search APIで各候補手をfork→自分の番を完了→ターン終了盤面を価値モデルで採点→最高評価を選択**（depth-1学習評価、0.3秒/戦＝高速）。
- [Measure] **対フィールド 価値bot 0.881 ≈ ヒューリスティック 0.905**（同条件84戦）。v1学習評価は競争力あるが僅差で未達。

## 5. 制約・課題
- [Fact] フィールドbotは薄い設定(25-30行)。[Hyp] **対フィールド0.90は現在のフィールド構成に依存している可能性**（強く回せば各対面の絶対値は動き得る）。
- [Fact] ただし遅い2進化デッキは**piloting非依存で速度負け**（[Measure] Dragapult ex着地が平均ターン7・47%）＝MegaStarmie優位の一部は本物。
- [Fact] **実ラダーで提出MegaStarmieはそれなりに負ける**（ユーザー観測）。[Hyp] **最有力かつ未検証**：simに雷デッキが無く弱点雷が未検証→実ラダーの雷アタッカー(例:メガライボルトex 200×2=400)に負けている可能性。
- 次フロンティア＝「学習評価(ベクトル化)の強化」or「LLMコーチ自動化」。

## 6. 検討してほしい論点（提出→研究の順）
1. **提出をどうする？**：(d) MegaStarmieの弱点(雷)対策・最終提出。雷弱点が実敗因か検証→デッキ調整(弱点軽減どうぐ/控えアタッカー)か別デッキかMS継続か。
2. **研究：価値モデル強化**：(a) カード埋め込み/MLP/均衡データで価値モデル自体を強くする（下記AlphaZeroの前提）。
3. **研究：公平なベンチマーク**：(c) フィールドbot強化で評価の妥当性を上げる。
4. **研究：LLMコーチ自動化**：(b) コーチ→提案→A/B→採用ループの自動化。

> 補足(GPT見解)：**AlphaZero型(探索＋方策ネット)は時期尚早**。価値モデルが0.881でヒューリスティック0.905に未達の段階では、「弱い評価関数×探索」では伸びない。**まず価値モデル単体を0.95近くに引き上げ、強い評価関数×浅い探索**を狙うのが筋。

## 8b. 汎用化実験の結果と研究仮説（重要）
- [Measure] **per-deck設定ゼロの汎用価値bot**で複数デッキを操作→**専用bot > 汎用Heuristic > 汎用価値bot**（例: Archaludon 0.60 / 0.50 / **0.30**、Dragapult 0.35 / 0.25 / **0.10**）。
- [結論] **問題は価値モデルでなく「状態表現」**。盤面が HP/サイド/エネ/ベンチ数 しか持たず、**「カードが何をできるか」が欠落**＝MegaStarmie/Archaludon/Dragapultが“同じ盤面”に見える。
- [設計方針(GPT)] カードIDのEmbedding単独は危険(新カードで再学習)。**カード→能力ベクトル**(HP/タイプ/進化段階/技数/最大ダメ/必要エネ/逃げエネ/特性有無/効果の構造)を特徴に。可変長盤面は**DeepSets**(各ポケをencode→平均→MLP、軽量・提出向き)。**順序: 状態表現 → 価値モデル → 探索**。
- **検証したい仮説**:
  - [Hyp A] 入力表現(カード能力特徴)を改善すると、汎用価値モデルは汎用Heuristicを超える。
  - [Hyp B] 探索の導入より、入力表現の改善の方が寄与が大きい。
  - [Hyp C] カード能力特徴だけで、新カード(未学習)へ汎化できる（固有名詞でなく「高火力・低コスト・効果無視のアタッカー」という概念を学ぶ）。
